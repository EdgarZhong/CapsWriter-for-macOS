# CapsWriter-Offline 协作规则

## 用户规则

- 永远用中文与用户沟通。
- 所有新增或修改代码必须附带详细中文注释。
- 当用户要求查看项目、总览项目、扫描目录时，必须递归查看目录结构后再总结。
- 当需求存在歧义时，不得自行脑补后直接开工，必须先收敛口径、给出建议、等待用户确认。
- 在执行代码修改、命令运行、文件移动前，必须先说明意图，再执行工具调用。

## 核心文档职责

- `readme.md`：只记录已经落实的稳定事实、运行入口、目录骨架、环境要求、关键文档索引。
- `AGENTS.md`：只记录通用规则、开发流程、协作边界、验收标准、文件安全策略。
- `CLAUDE.md`：只记录当前阶段目标、进度同步、任务看板、阶段决策、风险与执行状态。

## 文档维护规则

- 缺乏信息时，优先通过 `readme.md` 的文档索引寻找已有资料。
- `docs/` 存放专项规格、专项 SOP、调试说明、测试说明；除非用户明确要求，不主动查看 `docs/archived/`。
- 用户在会话中确认的口径收敛，必须尽快同步到对应文档。
- 新会话首轮必须查看项目内 `AGENTS.md` 与 `CLAUDE.md`。
- 新会话首轮必须用 `git` 查看近两次 Markdown 文档变更，避免偏离最新口径。

## 文件安全规则

- 永远不要删除文件。
- 需要废弃的文件一律移动到项目根目录 `.archive/`。
- `.archive/` 必须保持 Git 忽略状态。
- 覆盖重要文件前，先在 `.archive/` 中备份。
- 禁止执行 `rm`、`git clean`、`git reset --hard`、`sudo`、强推历史等破坏性操作。

## 代码规范

- 优先保持现有架构稳定，在此基础上做最小必要改动。
- 涉及平台差异时，优先新增抽象层或平台实现，不要把条件分支散落到业务流程各处。
- 所有平台相关能力至少覆盖以下维度：快捷键监听、文本注入、剪贴板/选中文本、前台窗口识别、推理后端选择。
- 修改配置项、运行方式、依赖方式时，必须同步更新文档入口。
- 新增注释必须解释设计意图、平台差异或非直观逻辑，避免只复述代码表面行为。
- 默认使用 ASCII；仅在中文文档、中文注释和已有中文文件中使用中文。

## 开发测试 SOP

1. 会话启动

- 查看 `AGENTS.md` 与 `CLAUDE.md`。
- 递归扫描项目目录，确认本次涉及的代码区域。
- 查看近两次 Markdown 提交，理解最新文档变化。

2. 方案收敛

- 对用户目标做简明复述。
- 在动手前说明本轮准备执行的命令或修改范围。
- 若存在高风险分支或多种技术路径，先给出推荐路径与理由。

3. 实施变更

- 优先使用仓库内现有结构和已有工具，不轻易引入新框架。
- 重要文件先备份到 `.archive/`。
- 大规模修改时按步骤推进，不一次性并发改很多区域。
- 运行 Git 长输出命令时统一使用 `git --no-pager`。

4. 验证闭环

- 先跑最小必要的定向验证，再决定是否扩大验证范围。
- 无法验证的内容必须明确说明原因、阻塞点和残余风险。
- 验证结果要回写 `CLAUDE.md` 的任务看板或状态区。

5. 收尾同步

- 稳定事实写回 `readme.md`。
- 规则、流程、验收要求写回 `AGENTS.md`。
- 阶段任务、状态、决策写回 `CLAUDE.md`。

## 通用验收要求

- 代码、配置、文档三者口径一致。
- 平台改造必须明确区分“已验证可用”和“仅完成代码接入”。
- 每次提交给用户的结果都要说明本轮完成项、未完成项、验证情况。
