# Qwen3-ASR macOS 专项规划

## 1. 文档目的

本文档只服务于 `mac-dev` 分支当前阶段目标：

- 仅支持 `Qwen3-ASR`
- macOS 服务端主路线采用 `qwen_asr_mlx`
- 保持现有 Client / Server 架构不做大重写
- 以后台常驻、离电使用、低发热、低延迟为优先目标

本文档不是长期全平台统一架构蓝图，也不是其他引擎的适配计划。

## 2. 当前结论

### 2.1 已确认事项

- `Qwen3-ASR` 是当前唯一保留的 ASR 路线。
- macOS 不再以现有 `qwen_asr_gguf` 为主后端。
- macOS 将新增 `qwen_asr_mlx` 后端。
- Windows 继续保留现有 `qwen_asr_gguf`。

### 2.2 采用 MLX 的原因

- 程序是后台常驻，不是偶发转录工具。
- 用户经常离电使用 Mac，更看重续航、发热与常驻成本。
- MLX 更贴近 Apple Silicon 的原生推理路径。
- 已有现成的 `mlx-qwen3-asr` 工作可以借鉴，不需要从零实现 Qwen3-ASR。

## 3. 范围收敛

### 3.1 本阶段明确要做

- 为 macOS 增加 `qwen_asr_mlx` 服务端后端。
- 让 CapsWriter 的客户端主链路在 macOS 上可用：
  - 全局快捷键触发录音
  - 麦克风录音采集
  - 识别结果文本注入
  - 必要的前台窗口识别
  - 必要的选中文字读取
- 首版只要求松开按键后尽快输出最终结果，不要求按住说话时展示中间结果流式文本。
- 选定一个默认 MLX 模型规格，并记录下载与切换策略。

### 3.2 本阶段明确不做

- 不适配 `SenseVoice`、`Fun-ASR-Nano`、`Paraformer`。
- 不把整个项目重构成 MLX-only。
- 不重做 UI，不做打包器，不先做发布流程。
- 不先追求所有高级能力一步到位，例如复杂角色联动或全量平台统一抽象。
- 不把“按住说话时的中间结果流式显示”纳入首版必做范围。

## 4. 方案对比结论

## 4.1 现有 `ONNX + GGUF` 路线

优势：

- 改动更小
- 复用现有代码更多
- 更快产出第一个“能跑”的版本

劣势：

- `encoder` 与 `decoder` 仍是两套运行时
- 更像兼容方案，不像 Mac 原生方案
- 对后台常驻、离电、低发热场景不占优

## 4.2 `qwen_asr_mlx` 路线

优势：

- 更贴近 Apple Silicon 原生推理路径
- 更适合常驻、短句高频触发和离电使用
- 已有现成实现可借鉴，避免从零造轮子
- 长期维护上更适合做 macOS 专用后端

劣势：

- 不是最小补丁
- 需要新增后端适配层
- 模型来源当前主要依赖社区 MLX 权重，而非 Qwen 官方直接维护的 ASR-MLX 发布物

## 4.3 最终决策

- macOS：采用 `qwen_asr_mlx`
- Windows：保留 `qwen_asr_gguf`
- 不做全局替换，只做按平台分流

## 5. 可借鉴的现有工作

## 5.1 直接可借鉴实现

- `mlx-qwen3-asr`
  - 已支持 `Qwen3-ASR`
  - 已提供 Python API
  - 已提供 `Session`
  - 已支持 `streaming`
  - 已支持 `timestamps`
  - 已提供 `mic`
  - 已提供 HTTP / OpenAI-compatible server

适合借鉴的方式：

- 不直接把它当成独立成品塞进 CapsWriter
- 而是把其 Python 侧能力适配到 CapsWriter 当前的 `BaseASREngine`

## 5.2 可获取的 MLX 模型权重

当前已确认可以较轻松获取的 MLX 格式 `Qwen3-ASR` 权重包括：

- `mlx-community/Qwen3-ASR-0.6B-bf16`
- `mlx-community/Qwen3-ASR-1.7B-bf16`
- `mlx-community/Qwen3-ASR-1.7B-4bit`

这意味着当前阶段**不需要先做自定义模型转换工具链**，可以先用现成模型验证后端接入。

## 6. 模型选型建议

## 6.1 选型原则

你的使用场景是：

- 后台常驻
- 高频短句语音输入
- 经常离电使用
- 目标设备是 Apple Silicon MacBook

因此优先级应该是：

1. 常驻成本
2. 发热与续航
3. 响应速度
4. 识别质量
5. 显存 / 内存占用

## 6.2 候选规格

### 方案 A：`Qwen3-ASR-1.7B-8bit`

特点：

- 质量高于 4bit 量化
- 在本地已经准备好权重的前提下，可以直接作为默认规格落地
- 常驻成本高于 4bit，但仍明显低于 `bf16`

判断：

- **在当前本地权重已经齐备的前提下，改为本阶段默认规格**

### 方案 B：`Qwen3-ASR-1.7B-4bit`

特点：

- 质量接近 1.7B 主规格
- 内存与磁盘占用明显低于 `bf16`
- 更适合后台常驻与离电使用

判断：

- **保留为本地回退规格**

### 方案 C：`Qwen3-ASR-1.7B-bf16`

特点：

- 质量上限最高
- 占用更大
- 常驻和离电成本更高

判断：

- 适合作为“插电 / 追求最高质量”的可选规格

### 方案 D：`Qwen3-ASR-0.6B-bf16`

特点：

- 更轻
- 更适合低内存占用场景
- 质量通常弱于 1.7B

判断：

- 适合作为轻量备用规格，不建议作为你的主规格

## 6.3 当前推荐

当前建议如下：

- 默认规格：`Qwen3-ASR-1.7B-8bit`
- 本地回退规格：`Qwen3-ASR-1.7B-4bit`
- 其他备选：`Qwen3-ASR-1.7B-bf16`、`Qwen3-ASR-0.6B-bf16`

## 7. 技术实施路线

## 7.1 服务端

### 目标

新增 `qwen_asr_mlx` 后端，而不是替换现有 `qwen_asr_gguf`。

### 首版结果策略

- 首版优先输出最终识别结果。
- 后端设计以“松开按键后快速完成最终识别”为核心目标。
- 不为了中间态流式回显增加额外推理与协议复杂度。

### 预计改动点

1. `config_server.py`

- 新增或调整 macOS 下的模型类型与模型目录配置
- 明确 `qwen_asr_mlx` 的默认模型规格

2. `core/server/engines/factory.py`

- 注册 `qwen_asr_mlx` 后端
- 保持原有 `qwen_asr` / `qwen_asr_gguf` 路线不被破坏

3. 新增 `core/server/engines/qwen_asr_mlx/`

- `asr_engine.py`
- `session_adapter.py` 或等价封装
- 必要的 schema / result 转换层

4. `requirements-server.txt`

- 为 macOS 增加 MLX 相关依赖
- 保持 Windows 依赖策略不被误伤

## 7.2 客户端

客户端仍按最小修改策略处理：

- 快捷键：平台分流
- 上屏：按平台分流
  - Windows 保留现有“默认模拟打字、命中 `Config.paste_apps` 时改走粘贴”的成熟策略
  - macOS 首版优先保证“复制到剪贴板 + 稳定粘贴上屏”的成功率
- 选中文字：改为按平台选择快捷键或可访问性方案
- 前台窗口：增强 macOS 检测可用性

### 结果返回后的上屏与剪贴板规格

当前阶段已确认的客户端体验口径如下：

#### 统一规则

- 松手结束录音后，客户端拿到最终识别结果时，应当：
  - 尝试把结果上屏到当前输入焦点
  - 同时把同一份最终结果复制到剪贴板
- “复制到剪贴板”是固定动作，不依赖上屏是否成功。
- “上屏”是额外动作，允许按平台采取不同实现策略。

#### Windows 口径

- 继续保留现有成熟体验，不因为 macOS 的适配问题而回退。
- 默认优先模拟打字上屏。
- 命中 `Config.paste_apps` 的应用时，强制走“剪贴板 + 粘贴”兜底。
- LLM typing mode 继续保留原有的流式打字体验。

#### macOS 口径

- 首版优先保证“最终结果稳定上屏”，不强求立即复刻 Windows 的流式打字观感。
- 允许默认采用“复制到剪贴板 + 粘贴上屏”的稳定路线。
- 待上屏、剪贴板和权限链路稳定后，再评估是否恢复更接近 Windows 的默认打字体验。

#### 当前实现关注点

- 结果输出链路需要与录音、识别链路分开看待：
  - 录音上传和服务端识别跑通，不代表上屏链路已经稳定
  - macOS 当前优先目标是先把“返回结果后不崩溃、能稳定上屏、剪贴板同步成功”做对
- 因此现阶段评估标准应优先是：
  - 松手后能拿到最终结果
  - 客户端不会在输出阶段崩溃
  - 当前前台输入框能收到结果
  - 剪贴板内容同步为最终结果

## 8. 分阶段计划

## 阶段一：文档与模型选型收敛

- 确认 MLX 主路线
- 确认默认模型规格
- 记录现成模型来源

完成定义：

- 文档口径统一
- 模型主规格明确

## 阶段二：服务端最小接入

- 新增 `qwen_asr_mlx`
- 跑通一次最小识别流程
- 先验证 file/audio -> text

完成定义：

- 服务端能在 macOS 上用 MLX 跑通 `Qwen3-ASR`

## 阶段三：流式与结果对齐

- 对齐 CapsWriter 当前最终结果结构
- 对接必要的时间戳能力
- 如有必要，仅保留内部增量处理，不把中间文本显示作为首版交付要求

完成定义：

- 新后端可以进入现有最终结果处理链路

## 阶段四：客户端主链路适配

- 快捷键触发录音
- 结果回注前台应用
- 权限与已知问题闭环

完成定义：

- 用户可以在 macOS 上完成“按键录音 -> 识别 -> 上屏”的主流程

## 9. 风险清单

### 风险一：MLX 后端接口与当前引擎抽象不完全一致

- 应对策略：增加薄适配层，不把第三方 API 直接散落到业务逻辑中

### 风险二：社区 MLX 权重与官方权重在版本节奏上可能不同步

- 应对策略：先固定当前可用模型版本，后续再补升级策略

### 风险三：macOS 权限限制影响输入链路

- 应对策略：把权限申请、验证步骤和已知限制写入文档

## 10. 验收标准

满足以下条件即可视为本阶段达标：

- 文档与代码口径明确“macOS = `qwen_asr_mlx`，Windows = `qwen_asr_gguf`”
- 能在 macOS 上加载默认 MLX 模型并完成识别
- 以“松开按键后尽快输出最终结果”为主要体验目标
- 客户端主输入链路可用
- 文档中明确默认模型、备选模型、已知风险与权限要求
